## CRUD処理/Create

CRUD処理を実装する際の基本手順
1.モデルクラスを作成する
2.CRUD処理をコントローラーにアクションとして定義する
3.アクションを実行するためのルーティングを設定する
4.ビューの作成を行う
5.コントローラーのアクションをそれぞれ実装する

**CRUD処理をコントローラーにアクションとして実装する**
```
rails g controller users index
```

---
**CRUD処理の基本の7つのアクション**
| アクション名 | アクションの用途 | HTTPメソッド |
| :-- | :-- | :-- |
| index | ユーザーを一覧表示する画面 | get |
| new | ユーザーを新規登録する画面 | get |
| create | ユーザーをDBに登録する処理 | post |
| show | ユーザー情報を確認する画面 | get |
| edit | ユーザー情報を編集する画面 | get |
| update | ユーザー情報を更新する処理 | patch |
| destroy | ユーザー情報を削除する処理 | delete |

---
**Railsのリンクはlink_toメソッド**
link_toメソッドの基本構文
```
<%= link_to "リンク名", url, data: {turbo_method: :HTTPメソッドの種類 } %>
```

**users_pathメソッドで指定(例)**
```
<%= link_to "ユーザー情報一覧", users_path %> 
```
**シンボルで指定**
```
<%= link_to "ユーザー情報一覧", :users %>
```
この記述によりメソッドの戻り値である`/users`がURLに指定されルーティングで設定されたアクションが実行される。


**form_withメソッドの基本構文**
```
<%= form_with model: モデルオブジェクト do |f| %>
  フォームの中身
<% end %>
```

例
```
<%= form_with model: @user do |f| %>
  <%= f.text_field :name %>
<% end %>
```
この場合のフォームがHTMLとして読み込まれると以下のようになる。
```
<input type="text" name="user[name]" id="user_name">
```

---
users_controller.rbのcreateアクションで重要となる箇所は以下のコード
```
@user = User.new(params.require(:user).permit(:name, :email, :age, :introduction))
```
この行ではフォームに入力した値をnewメソッドで作成したUserモデルのインスタンスに渡している。paramsメソッドはフォームで送信されるデータを取り出すことが出来るメソッド。paramsメソッドにはrequireメソッドとpermitメソッドを指定することができる。
requireメソッドではデータのオブジェクト名を指定する。この時に指定したオブジェクト以外であれば例外が発生する。
permitメソッドは指定した「キー」を変更可能と許可することができる。
requireとpermitメソッドを使用すっ理由はユーザーなどから予期しない構造で渡ってきたパラメーターに対してDBに登録する前の段階で例外とすることを目的としている。DBに登録の際のセキュリティー向上のためと考えればOK。

---
**saveメソッドでDBにデータを登録する**
```
@user.save
```
@userにはフォームで入力した値が設定されているUserモデルのインスタンスが格納されており、それに対してsaveメソッドを実行することでUserモデルのその値をDBに新しいレコードとして保存することができる。
また、saveメソッドの戻り値はDBに登録が成功したらtrueを、それ以外はfalseを返す。これを利用してDBに登録できた場合と出来なかった場合の処理をif文と組み合わせて実装することもできるというわけだ。

---
**redirect_toメソッドで別のURLで再アクセスさせることができる**
コントローラーのアクションが終了すると実行したアクション名と同じ名前のビューファイルが参照される。しかし、今回のcreateアクションではDBに登録する処理の実装だった。この登録が完了したらここではユーザーの一覧画面を表示するほうがWebサイトとしては自然な流れとなる。RailsではURLへのアクセスを強制的に変更できるredirect_toメソッドがある。

**redirect_toメソッドの基本構文**
```
redirect_to リダイレクト先のURL
```

**renderメソッドでビューファイルを表示する**
@user.saveが失敗した場合の処理についても確認する。saveメソッドがfalseを返した場合、ユーザーにはエラーの内容を表示して再度入力してもらう必要がある。このような場合に使用するのがrenderメソッド。
renderメソッドは指定したビューファイルを表示するためのメソッド。redirect_toメソッドとは異なり、URLを変更することなく別のビューファイルを表示することが出来る。

**renderメソッドの基本構文**
```
render "ビューファイル名", status: :ステータスコード
```

例
```
render "new", status: :unprocessable_entity
```

---
**HTTPステータスコード422の重要性**
:unprocessable_entityは「処理できないエンティティ」を意味するHTTPステータスコード422に対応する。Rails7では、このステータスコードを指定することが非常に重要。

**フラッシュメッセージの定義の仕方**
```
flash[:キー] = フラッシュメッセージとして格納するオブジェクト
```

フラッシュメッセージとはリダイレクト時などで一度だけ表示できる機能のこと。ユーザーの登録完了や削除などの時にデータに変更があったことがメッセージとして表示できると原理。

フラッシュメッセージはハッシュとして格納され、格納したオブジェクトは`flash[:キー]`取り出すことができる。

---
**findメソッドとは**
| メソッド | 基本構文 | 使用例 | 使用例の意味 |
| :-- | :-- | :-- | :-- |
| find | モデル名.find(id) | User.find(1) | Userモデルからidが「1」のレコードを読み込む |
| find_by | モデル名.find_by(カラム名:値) | User.find_by(name:"hoge") | Userモデルのnameカラムから値が`hoge`である最初のレコードを一つ読み込む |
| where | モデル名.where(カラム名:値) | User.where(age:20) | Userモデルのageカラムから値`20`であるレコードを全て読み込む |
| all | モデル名.all | User.all | Userモデルから前レコードを読み込む |

**find_byメソッドとwhereメソッドの違い**
- `find_by`メソッドは指定したカラムから指定した値を持つレコードの「最初の一件」を読み込む
- `where`メソッドは指定したカラムから指定した値を持つレコードを「全て」読み込む

**paramsメソッドとは**
`params`メソッドは送られてきたパラメータを受け取ることができるメソッド。リンク、フォーム、配列やハッシュなどのパラメータを受け取ることが出来る。
